## 1. [Critical] Vault-сентинелы видны в форме для настоящих секретов

**Affected UI:**

- `apps/web/src/components/proxies/WireGuardForm.vue` — поля `private-key` и `pre-shared-key`.
- `apps/web/src/components/profile/ProfileForm.vue` — поле `secret` (mihomo Bearer-token на "Панели управления").

**Что видит пользователь (скриншоты):**

- "Приватный ключ" → `$MIHARBOR_VAULT:09e0bb8a-acf0-4953-a75f-0e9fd2146a0d` + ошибка "Некорректный WireGuard-ключ (base64, ≥40 символов)".
- "Общий ключ (опционально)" → `$MIHARBOR_VAULT:12c35938-0b7d-47d3-aacc-b73fe27f5707`.
- "Секрет (Bearer-токен)" → `$MIHARBOR_VAULT:a45435ca-69a6-4665-a6e1-5de955e3789d`.
- Пользователь не может отредактировать: либо validator отвергает, либо пользователь боится что в конфиг сохранится сентинел вместо реального значения.

**Корневая причина.** И WireGuardForm, и ProfileForm получают свои начальные значения из **draft-derived** данных — `parseDraft(draftText)` → `getProfileConfig()` / `draftProxies` — где секреты всегда vault-masked как `$MIHARBOR_VAULT:<uuid>`. View-специфичные JSON endpoint'ы (`/api/config/meta`, `/api/config/proxies`) отдают **фиксированные сентинелы** (`META_SECRET_SENTINEL`, `WIREGUARD_PRIVATE_KEY_SENTINEL`, `WIREGUARD_PRE_SHARED_KEY_SENTINEL`), которые UI умеет обрабатывать (disable reveal, round-trip unchanged), но формы не смотрят в эти endpoint'ы.

Наш v0.2.5 Section 3 фикс решил эту проблему для `public-key` за счёт вывода его из vault scope (`KNOWN_NON_SECRET_KEYS`). Но `private-key`, `pre-shared-key`, `secret` — это **настоящие секреты**, их нельзя просто выкинуть из vault'а.

**Fix plan:**

1. `packages/shared/src/types/sentinels.ts`:
   ```ts
   export const VAULT_SENTINEL_PREFIX = '$MIHARBOR_VAULT:'
   export function isVaultSentinel(v: unknown): boolean {
     return typeof v === 'string' && v.startsWith(VAULT_SENTINEL_PREFIX)
   }
   ```
2. `WireGuardForm.vue`: расширить computed'ы:
   ```ts
   const privateKeyIsSentinel = computed(
     () => privateKey.value === WIREGUARD_PRIVATE_KEY_SENTINEL || isVaultSentinel(privateKey.value),
   )
   const preSharedKeyIsSentinel = computed(
     () =>
       preSharedKey.value === WIREGUARD_PRE_SHARED_KEY_SENTINEL ||
       isVaultSentinel(preSharedKey.value),
   )
   ```
   И `privateKeyError` должен вернуть `null` когда `privateKeyIsSentinel.value === true` (скип валидации). Аналогично `publicKeyError` — там public-key после v0.2.5 не должен быть vault-сентинелом, но на всякий случай тот же скип для чистоты.
3. `ProfileForm.vue`: расширить `secretIsSentinel`:
   ```ts
   const secretIsSentinel = computed(
     () =>
       props.modelValue.secret === META_SECRET_SENTINEL || isVaultSentinel(props.modelValue.secret),
   )
   ```

**Round-trip безопасен.** Если пользователь не меняет значение — сентинел остаётся в draft'е → на deploy `vault.unmaskDoc()` резолвит `$MIHARBOR_VAULT:<uuid>` обратно по uuid'у в on-disk payload. Это существующий механизм.

**Тесты:**

- `packages/shared/tests/sentinels.test.ts` — позитивные/негативные для `isVaultSentinel`.
- `apps/web/tests/proxies.spec.ts` — добавить кейс: `initial['private-key'] = '$MIHARBOR_VAULT:<uuid>'` → форма без ошибок, reveal-eye disabled, submit round-trip'ит неизменённым.
- `apps/web/tests/profile.spec.ts` — тот же кейс для `secret`.

## 2. [Important] SSE `/api/health/stream` падает с `ERR_HTTP2_PROTOCOL_ERROR 200 (OK)`

**Consoled повторно:**

```
GET https://miharbor.thematrix.su/api/health/stream net::ERR_HTTP2_PROTOCOL_ERROR 200 (OK)
```

**Корневая причина.** В `apps/server/src/routes/sse.ts:69-75` и `:100-107` возвращается header `Connection: keep-alive`. По RFC 7540 §8.1.2.2 этот заголовок **запрещён в HTTP/2** (connection-specific). Reverse-proxy перед thematrix.su (вероятно Caddy или nginx) или сам браузер рвёт стрим с `ERR_HTTP2_PROTOCOL_ERROR` сразу после получения первых байт.

**Последствие.** Health-badge в Header'е не обновляется через SSE; работает только fallback-polling (если он есть — нужно проверить `useHealthBadge` composable). Консоль спамит retry-попытки.

**Fix.** Удалить `connection: 'keep-alive'` из обоих `Response` headers в `apps/server/src/routes/sse.ts`. Keep-alive — дефолт в HTTP/1.1 persistent connections и в HTTP/2, явно объявлять не надо.

```ts
// sse.ts:68-75 → убрать строку с connection
return new Response(stream, {
  headers: {
    'content-type': 'text/event-stream; charset=utf-8',
    'cache-control': 'no-cache, no-transform',
    // connection: 'keep-alive', ← удалить
    'x-accel-buffering': 'no',
  },
})
```

**Тесты:**

- `apps/server/tests/routes/` — смок на GET `/api/health/stream`, проверить что ответ не содержит `connection` header.
- Manual: открыть страницу в проде после деплоя, убедиться что в DevTools → Network вкладка `/api/health/stream` держится как `pending` (стрим), без 4×retry.

## 3. [Minor] Chrome DOM warning: "Multiple forms should be contained in their own form elements"

**Console:**

```
[DOM] Password field is not contained in a form: <input id="profile-secret" type="password" ...>
[DOM] Multiple forms should be contained in their own form elements; break up complex forms into ones that represent a single action: <form class="space-y-4" data-testid="wireguard-form">
```

**Корневая причина.** Chrome autofill эвристика: видит два `type="password"` поля (private-key + pre-shared-key) в одной `<form>` у WireGuardForm + считает Amnezia-секцию с `<button type="button">` внутри формы "вторичным action". Аналогично ProfileForm с полем `secret` без `<form>` обёртки (оно висит прямо в вёрстке Profile.vue).

**Последствие.** Косметическое предупреждение. На submit/валидацию не влияет. Safari/Firefox молчат.

**Fix:**

- WireGuardForm: добавить `autocomplete="off"` на `<form>`:
  ```vue
  <form
    class="space-y-4"
    data-testid="wireguard-form"
    autocomplete="off"
    @submit.prevent="onSubmit"
  ></form>
  ```
- ProfileForm: если поле `secret` на странице не обёрнуто в `<form>`, добавить минимальную обёртку `<form autocomplete="off" @submit.prevent>` либо явно поставить `autocomplete="new-password"` на сам input.

## 4. [Important] Версия конфига не показывается в Header'е — "Конфиг v—"

**Скриншот.** Бейдж в центре хедера показывает "`Конфиг v—`" вместо реального номера версии.

**Корневая причина.** `Header.vue:37-40` берёт версию так:

```ts
const configVersion = computed<string>(() => {
  const version = config.meta?.version ?? config.meta?.revision ?? '—'
  return t('header.config_version', { version })
})
```

Оба поля `version` и `revision` приходят из `/api/config/meta` (`apps/server/src/config/views/meta.ts`). Если в mihomo config.yaml этих полей нет на верхнем уровне — бейдж показывает `—`.

**Вопрос.** Нужно ли:

- (a) убрать бейдж полностью когда версия не задана (чистое решение);
- (b) вывести другое полезное значение (например, `git rev-parse --short HEAD` последнего snapshot'а);
- (c) добавить в mihomo конфиг поле `version: '0.2.5'` и документировать;
- (d) считать хэш config.yaml и показать первые 8 символов как fingerprint?

Вариант (d) или (a) разумны. Обсудить с пользователем.

**Note.** Эта проблема **не регрессия v0.2.5** — тот же бейдж висел в v0.2.4 с дефисом, просто пользователь заметил только сейчас.

## 5. [Important] Странные переносы строк в History → Snapshot Diff viewer

**Скриншот.** На странице `/history` при открытии первого snapshot'а (с пометкой "применил canonicalization") diff рендерится с визуальными разрывами строк: префикс `+` стоит на одной строке, содержимое `# mihomo config...` — на следующей, между ними пустая строка. Итого каждая строка diff'а занимает 2-3 визуальных строки, что делает снапшот огромного конфига (1423 строк) нечитаемым.

**Affected:** `apps/web/src/components/history/SnapshotDiffDrawer.vue`. Пользователь находится на роуте `/history`, открывает первый snapshot → drawer показывает `diff.patch` из `/api/snapshots/:id` через `diff2html` render.

**Гипотезы причин:**

1. **CSS-проблема.** В `SnapshotDiffDrawer.vue:169+` есть минимальный subset diff2html стилей. Возможно `.d2h-code-line` получает `display: block` вместо `inline` для префикса. Скорее всего это — нужно проинспектировать DOM в DevTools.
2. **`line-by-line` vs `side-by-side` mismatch.** Код вызывает `html(patch, { outputFormat: 'line-by-line' })` — это должно рендерить каждую строку diff'а в одной таблично row с префиксом и содержимым в разных `<td>`. Если diff2html вдруг рендерит иначе (регрессия пакета) или наш CSS override ломает table-cell → block.
3. **Synthetic diff.** `buildSyntheticDiff` в `SnapshotDiffDrawer.vue:105-116` собирает первый snapshot как `---previous\n+++snapshot <id>\n@@ -0,0 +N @@\n <content>`. Каждая строка `content` предваряется пробелом как context. Но на скриншоте строки начинаются с `+`, т.е. патч пришёл от сервера реальный (не synthetic). В snapshot.ts при первом snapshot'е `createTwoFilesPatch('/dev/null', id, '', maskedText, ...)` генерирует `+<line>` для всех строк — это корректно.
4. **Бага нашего v0.2.5 `PendingChangesDialog.vue`.** Его `<style scoped>` с `:deep(.d2h-*)` селекторами мог утечь на SnapshotDiffDrawer (unlikely из-за `scoped`, но проверить CSS сборки). Возможно Tailwind purge включил классы из PendingChangesDialog, и они теперь глобальные где-то.

**Отладка следующей сессии:**

- Открыть DevTools на /history, посмотреть computed styles на `.d2h-code-line` и `.d2h-code-linenumber`.
- Проверить raw `diff.patch` ответа `/api/snapshots/:id` — там должен быть `+# =========...`, а не `+\n# =========...`.
- Сравнить с рендером PendingChangesDialog (new v0.2.5 feature): выглядит ли там diff так же странно? Если да — bug в общем diff2html CSS; если нет — scoped-style leak.

**Fix direction.** Зависит от находок отладки.

## 6. [Feature] Селектор RULE-SET из rule-providers

**Запрос.** Пользователь хочет чтобы для типа правила `RULE-SET` в RuleEditor появлялся combobox со списком настроенных rule-provider'ов, аналогично v0.2.5 GEOSITE/GEOIP selector.

**Источник данных.** `profile.rule-providers` секция mihomo конфига (на верхнем уровне YAML, структура `{<name>: {type, behavior, url, ...}}`). Имена rule-провайдеров — это ключи этого dict'а.

**Бонус (необязательно).** Mihomo REST API `GET /providers/rules` (уже обёрнуто в `apps/server/src/mihomo/api-client.ts:184` → `listRuleProviders`). Преимущество: видны не только YAML-декларированные, но и динамически подгруженные rule providers. Недостаток: зависит от живого mihomo, при выключенном — combobox пустой.

**Рекомендую: читать из live config** (как `geo-source.ts` читает `geox-url`). Если rule-providers не заданы — dropdown пустой, free-form input всё равно работает.

**План:**

1. `apps/server/src/routes/catalog.ts` — расширить `GET /api/catalog/geo` → `GET /api/catalog/rule-providers` (или добавить третью секцию в существующий endpoint). Возвращать `{ names: string[], source: 'profile.rule-providers' }`.
2. `apps/web/src/stores/catalog.ts` — добавить `ruleProviders: string[]` + `loadRuleProviders()`.
3. `apps/web/src/components/services/GeoCatalogCombobox.vue` → переименовать в `CatalogCombobox.vue`, параметризовать `type: 'GEOSITE' | 'GEOIP' | 'RULE-SET'` и `source: 'geosite' | 'geoip' | 'rule-providers'`. Либо: сделать новый отдельный `RuleSetCombobox.vue`, не мешать с гео-комбо.
4. `RuleEditor.vue` — расширить `isGeoType` → `isCatalogType` включающий `RULE-SET`, подбирать правильный catalog по `type.value`.

Отдельный компонент проще — разные источники, разные UX (rule-providers обычно 5-20 штук, плоский список без fuzzy-search нужен меньше).

**Тесты:**

- Server: `routes/catalog.test.ts` — добавить кейс на `rule-providers` parsing + fallback когда секция отсутствует.
- Web: новый `rule-set-combobox.spec.ts` + расширить `rule-editor-geo.spec.ts` → `rule-editor-catalog.spec.ts` с RULE-SET случаем.
